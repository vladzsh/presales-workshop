# Utility Tree (Дерево корисності)

**Utility Tree** — це фундаментальний ієрархічний інструмент аналізу, який використовується для ідентифікації, деталізації та **пріоритезації вимог до атрибутів якості ([[Якість вимог (NFR)|NFR]])**.

## Механіка та структура Utility Tree

Інструмент дозволяє перетворити абстрактні бізнес-цілі на конкретні технічні сценарії (Architecturally Significant Requirements — ASR). Структура дерева має чотири рівні:

1. **Utility:** Загальна якість системи.
2. **Quality Attribute:** Основні категорії (наприклад, Performance, Security, Availability).
3. **Attribute Refinement:** Уточнення атрибута (наприклад, для Performance — Response Time або Throughput).
4. **Scenario:** Конкретний вимірюваний сценарій (наприклад: «Результати пошуку відображаються менш ніж за 2 секунди при 500 одночасних користувачах»).

## Пріоритезація атрибутів (Матриця 2x2)

Ключовою функцією Utility Tree є об'єктивізація вибору архітектурних рішень через пріоритезацію сценаріїв за двома шкалами:

- **Business Value (Бізнес-цінність):** Наскільки цей сценарій критичний для успіху продукту (High/Medium/Low).
- **Architectural Impact (Архітектурний вплив):** Наскільки складно або дорого реалізувати цей сценарій технічно (High/Medium/Low).

Фокус має бути на сценаріях типу **(H, H)** — тих, що мають високу цінність і високу складність, оскільки саме вони визначають основні архітектурні ризики та вартість проекту.

## Utility Tree у контексті інших інструментів

Дерево корисності є «вхідною точкою» для всього подальшого технічного аналізу:

- **Зв'язок з [[ATAM]] (Tradeoff Analysis):** Utility Tree виявляє конфліктуючі атрибути (наприклад, Performance vs. Security). Після того, як пріоритети розставлені, архітектор використовує ATAM для прийняття інформованого рішення про компроміс.
- **База для естимації:** Конкретизація вимог у дереві дозволяє уникнути пастки розмитих запитів, що веде до більш точних часових та фінансових оцінок.
- **Управління ризиками (RAPID):** Сценарії з високим архітектурним впливом автоматично стають кандидатами до логу ризиків.
- **Візуалізація архітектури ([[C4 Model]]):** Пріоритезовані в дереві атрибути визначають, які саме компоненти системи потребують детальнішої ілюстрації.

## Роль у комунікації та «Технічній совісті»

Використання Utility Tree допомагає архітектору виконувати роль **«технічної совісті»** угоди:

- **Створити спільне розуміння з клієнтом:** Замість суперечок про суб'єктивні терміни, сторони погоджують вимірювані сценарії.
- **Едукація стейкхолдерів:** Архітектор пояснює, чому сценарій з пріоритетом (H, H) вимагає більшого бюджету або іншого технологічного стеку.
- **Blue-print для команди:** Задокументоване дерево стає частиною «контракту» на поставку, що дозволяє команді імплементації не вигадувати вимоги заново.

> **Utility Tree** — це дисциплінуючий інструмент, який перетворює хаотичні побажання замовника на структурований технічний беклог, що є критично важливим для успішного проведення Discovery та якісної підготовки Proposal.
