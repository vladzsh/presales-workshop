# C4 Model

У межах **архітектурного підходу** під час фаз Pre-sale та Discovery модель **C4** розглядається як найбільш ефективний стандарт для візуалізації технічного рішення. Її головна перевага полягає в ієрархічності та здатності «зумувати» складність системи для різних аудиторій.

## Рівні C4

### L1: System Context
Погляд «з висоти пташиного польоту» на технічне рішення.

- **Система як «чорна скринька»:** На цьому рівні система відображається як єдиний центральний елемент, без розкриття її внутрішньої структури.
- **Взаємодія з оточенням:** Діаграма фокусується на тому, як система комунікує з кінцевими користувачами (акторами) та іншими зовнішніми системами (платіжні шлюзи, CRM тощо).
- **Цільова аудиторія:** Ідеальний для нетехнічних стейкхолдерів та **C-level менеджменту**.
- **Уникнення «змішування рівнів»:** Змушує архітектора дотримуватися одного рівня абстракції.

### L2: Container Diagram
Критичний етап деталізації, який дозволяє «зумувати» систему.

- **Внутрішня структура:** Відображає високорівневі технічні блоки: веб-додатки, SPA, мобільні застосунки, бази даних, файлові сховища та API.
- **Технологічний стек:** Саме на цьому рівні стає зрозумілим обраний стек (React, Python/Django, Postgres тощо), що є ключовим для подальшої естимації.
- **Presale-стандарт:** Разом із L1 складає «гігієнічний мінімум» технічної документації для Proposal.
- **Основа для TCO:** Дозволяє ідентифікувати всі необхідні хмарні ресурси та бази даних для розрахунку **Cost of Ownership**.

### L3: Component Diagram
Глибока деталізація окремого контейнера.

- **Вміст:** Відображає, як контейнер розбитий на окремі компоненти (контролери, сервіси, внутрішні модулі).
- **Мікросервісна декомпозиція:** Найкраще описувати саме на цьому рівні.
- **Застосування:** Зазвичай використовується під час **платних Discovery**, а не швидких пресейлів.
- **Аудиторія:** Переважно для розробників та архітекторів — безпосередній "blueprint" для імплементації.

### L4: Code
Четвертий і найбільш деталізований рівень.

- **Суть:** Класичне UML-документування (діаграми класів), що показує внутрішню логіку компонентів.
- **У Pre-sale/Discovery:** Майже завжди **ігнорується** — занадто детальний для етапів продажу та дослідження.
- **Сучасний підхід:** З поширенням автоматичної генерації документації з коду (Swagger, інлайн-документування) архітектори часто відмовляються від мануального створення L4-діаграм.

## Роль C4 в архітектурному підході

1. **Подолання хиби змішування рівнів:** C4 змушує архітектора чітко розмежовувати рівні абстракції, що запобігає поширеній помилці — коли на одній схемі малюють і кінцевого юзера, і специфічний протокол зв'язку між мікросервісами.
2. **Створення «Blueprint» для команди:** Документація стає фактичним контрактом на поставку та основою для майбутнього Software Design Document (SDD).
3. **Еволюційний розвиток (Gardening):** Архітектурна документація не є статичною; вона починається з начерків на пресейлі та постійно «доглядається» і деталізується впродовж усього життєвого циклу проекту.
4. **Використання інструментів:** Для швидкості архітектори часто використовують підхід _Diagrams as Code_ (наприклад, через Structurizr), що дозволяє легко адаптувати шаблони під нові проекти.
